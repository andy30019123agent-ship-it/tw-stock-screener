## 3. 成本/風控（FinMind 額度）

### 🟡 中 — 手動觸發 + 排程同時跑，可能雙倍消耗 API 額度
- **位置**：`.github/workflows/daily.yml` 第 16~18 行
- **問題**：`concurrency.cancel-in-progress: false` → 若手動觸發（workflow_dispatch）與排程（cron 18:17）在同一分鐘內重疊，兩個 workflow 都會跑完整流程，消耗 2×226 = 452 次 FinMind 請求。
- **現況**：112 支股票 × 2 次/股 + 2 次管理請求 = 226 次/天，免費版 600次/hr，正常不超限。雙倍仍在 600 次內，目前風險低，但若清單擴大（上櫃股加進來可能到 300+ 支）就有超限疑慮。
- **怎麼修**：改 `cancel-in-progress: true`，或加 `timeout-minutes: 30` 防止異常長跑。

### 🟡 中 — 超限後靜默返回空資料，前端警告依賴 `expected` 欄位
- **位置**：`pipeline/build_data.py` 第 51~55 行
- **問題**：額度超限時 `api_get()` 最多重試 3 次（各 sleep 8s），然後 `return []`。後續 `compute_indicators()` 因 `len(closes) < 65` 返回 None，該股被略過，`results` 少了這些股。前端靠 `expected vs count < 90%` 顯示橘色警告，但若超限僅影響 5~10% 的股票，警告不會觸發，使用者不會察覺資料不完整。
- **怎麼修**：加計數器統計被略過的股票數，在最後輸出時加進 JSON：`"api_failures": N`；或改設門檻（例如 80%）。

- 無快取代表每天 build 都消耗完整 API 額度。但目前免費版足夠，建議列為下一版優化。
